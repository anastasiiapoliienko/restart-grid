Project brief / Engineering plan

RL-learned cold-load pickup for the Ukrainian grid

An advisory system that learns the safest sequence to re-energize feeders after a blackout — trained on a high-fidelity twin of Ukrainian topology and seasonal load, validated in hardware-in-the-loop, deployed shadow-mode beside operators at Ukrenergo and DSOs.

scope sub-transmission + distribution restoration horizon 24 months to closed-loop advisor team 6–10 (RL, power systems, ops, frontend) budget ~€2.4M phased

1 · The problem in one paragraph

After a national or regional blackout, restoring a feeder is not "flip the breaker." With every thermostat, fridge, and heater off, load diversity is lost: the moment power returns, every device demands current at once, producing inrush 3–8× the steady-state load. The upstream breaker trips, and you black out again. Operators currently follow paper procedures that assume seasonal averages. Ukraine, in 2026, has neither average seasons nor static topology — substations are damaged, repaired, and damaged again on a weekly cadence, and restoration sequences written six months ago are no longer valid.

pain

Lost diversity

All thermostatic loads call simultaneously after a cold outage. Inrush dwarfs nameplate demand.

pain

Topology drift

Substations and lines change weekly. Static procedures go stale faster than they can be rewritten.

pain

Operator cognitive load

Restoring under attack with degraded telemetry is the worst possible time to do paper procedures from memory.

2 · Why this is an RL problem, not an optimization one

Classical CLPU studies frame restoration as a constrained optimization: pick segment order to minimize a cost. That works when the system is well-modeled and stationary. Ukraine's restoration problem is none of that — load behavior after a 12-hour winter outage is non-linear, topology is partially observed, and the "right" action depends on weather forecast, time since outage, ambient temperature, and which neighboring feeders are already loaded. The mapping state → next action is exactly what RL learns well, and is a poor fit for solver-based approaches that require re-formulating every time the grid changes shape.

3 · How it will be built — six steps

  1. Digital twin of a real Ukrainian sub-region. Start with one oblast's 35 kV / 10 kV network in OpenDSS or GridLAB-D. Calibrate from historical SCADA traces (Ukrenergo + a partner DSO). Model thermostatic loads explicitly with a population of equivalent thermal parameter (ETP) models so cold-load behavior emerges, not hand-coded.
  2. Scenario generator. Synthesize blackouts conditioned on: outage duration, ambient temp, time-of-day, damage pattern (single substation, multi-node, partial line). Sample millions of restoration starting states. This is where the project lives or dies — the agent will only be as good as the diversity of scenarios.
  3. RL environment. State: per-segment energization flag, ambient temperature, outage duration, load forecast vector, breaker headroom, neighbor-feeder status. Action: choose next segment to close, voltage ramp rate, dwell time before next action. Reward: +load restored − λ·trips − μ·time-to-99%, with a hard penalty for cascading reclosure failures.
  4. Train a PPO/SAC agent in Stable-Baselines3 or Ray RLlib. Curriculum: small networks first, increasing topology size and damage severity. Save policies with explainability hooks — every recommended action carries a SHAP-style breakdown of which state features drove it.
  5. Hardware-in-the-loop validation. Replay the trained policy against a real-time digital simulator (RTDS or Typhoon HIL) running the same network at millisecond resolution. This is the sim-to-real check: does the policy still beat baseline when ETP parameters are perturbed and measurement noise added?
  6. Deploy shadow-mode alongside operators at one DSO. The agent's recommendation appears next to the operator's planned action; operator decides. Telemetry from real restorations becomes training data. Only after 6+ months of agreement does the system promote to advisory-with-approval (operator clicks "accept"), and never to fully closed-loop.

6 · Reward function (concrete)

# per environment step
reward = (
    1.0  * delta_load_restored_pu      # MW reconnected this step, p.u.
  - 25.0 * breaker_trip                 # hard penalty
  - 0.05 * step_seconds                 # mild urgency pressure
  - 2.0  * voltage_violation_pu         # V outside [0.93, 1.05]
  - 0.5  * thermostat_diversity_loss    # proxy for compounding inrush risk
)
# terminal bonus when ≥99% load restored without trips
if done and restored >= 0.99 and trips == 0:
    reward += 50.0

7 · State and action spaces

State (per step)

  • energized[N] — boolean per segment
  • cold_minutes[N] — outage duration per segment
  • load_forecast[N, H] — H-step ahead per segment
  • ambient_C, weather_24h
  • upstream_headroom_pu
  • neighbor_state[K] — adjacent feeder energization
  • topology_hash — current network config ID

Action (per step)

  • segment_id — which to energize next (or no-op)
  • ramp_seconds ∈ {0, 5, 15, 60} — voltage ramp
  • dwell_seconds ∈ {0, 30, 120, 600} — wait before next action

Discrete + small: ~8 × 4 × 4 = 128 actions per step. Manageable for PPO.

8 · Data we need (and how to get it)

required

SCADA history, ≥18 months

From Ukrenergo + at least one DSO (DTEK Grids, Lvivoblenergo). Hourly load and switching events per substation. NDA required; data stays on-prem.

required

Network topology (CIM/CGMES)

Static model in CIM. Versioned snapshots so the agent's topology_hash input is meaningful.

helpful

Strike / damage log

Lets us train on realistic damage patterns rather than uniform random faults.

helpful

Smart-meter aggregates

Even regional aggregates help calibrate ETP load model parameters seasonally.

10 · Phased roadmap

phase 0 — 3 months

Single-feeder proof

  • One 10 kV feeder in OpenDSS
  • Synthetic scenarios only
  • Show RL beats greedy + paper procedure on time-to-99% and trip count
phase 1 — months 4–9

Regional twin

  • Full oblast network calibrated to historical SCADA
  • Curriculum training across seasons and damage patterns
  • Explainability hooks shipped
phase 2 — months 10–18

HIL + shadow mode

  • Typhoon HIL validation
  • Console deployed at one DSO; operator sees agent's recommendation but is not bound by it
  • Disagreement log feeds retraining
phase 3 — months 19–24

Advisory-with-approval

  • Operator clicks accept; SCADA executes
  • Two-person confirmation for actions above a learned-risk threshold
  • Never fully closed-loop in v1

11 · Risks and mitigations

risk

Sim-to-real gap

ETP load behavior in the field will not match the population we sampled. Mitigation: domain randomization during training, plus online fine-tuning from shadow-mode telemetry.

risk

Operator distrust

If the agent ever recommends something visibly wrong, trust is gone for a year. Mitigation: always show why (SHAP), never override, ship long shadow period before approval mode.

risk

Topology changes invalidate policy

A re-routed substation is a different MDP. Mitigation: condition the policy on topology_hash and meta-train across a distribution of topologies so adaptation is fast.

risk

SCADA data access

The single highest-risk dependency. Mitigation: start with synthetic data + one DSO's anonymized historical exports; widen access after phase-0 results land.

12 · Success metrics

  • Time-to-99% load restored, vs. operator baseline, on held-out scenarios. Target: −25% by phase 1.
  • Cascading trips per restoration on the twin. Target: <0.1 average.
  • Operator-agreement rate in shadow mode. Target: ≥80% by month 15.
  • Decisions per minute the operator can confidently make. Soft target: +30%.
